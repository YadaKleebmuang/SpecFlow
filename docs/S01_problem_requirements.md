# S01: เอกสารวิเคราะห์ปัญหาและความต้องการ (Problem Analysis and Requirements Specification)

---

## 1. ปัญหาที่พบ (Problem Statement)
ในการจัดสเปคหรืออัปเกรดเครื่องคอมพิวเตอร์ส่วนบุคคล (Personal Computer - PC) ผู้ใช้งานทั่วไปมักประสบปัญหาสำคัญหลายประการอันเนื่องมาจากความซับซ้อนเฉพาะทางของอุปกรณ์ฮาร์ดแวร์ไอที:

1. **ความยากในการเลือกอุปกรณ์สำหรับผู้ไม่มีความรู้ทางเทคนิค:** ผู้ใช้มักไม่ทราบว่าความต้องการใช้งานของตนเอง (เช่น การเล่นเกม, งานตัดต่อกราฟิก หรือการพิมพ์เอกสารทั่วไป) สอดคล้องกับอุปกรณ์ชิ้นใด และการกระจายงบประมาณที่เหมาะสมควรเป็นอย่างไร
2. **ปัญหาความเข้ากันไม่ได้ของอุปกรณ์ (Hardware Compatibility Issues):** การจัดสเปคคอมพิวเตอร์ประกอบมีความเสี่ยงสูงที่จะเลือกชิ้นส่วนที่เข้ากันไม่ได้ทางกายภาพ เช่น เลือกหน่วยประมวลผลกลาง (CPU) และเมนบอร์ด (Mainboard) ที่มีรูปแบบ Socket ต่างกัน หรือการเลือกใช้หน่วยความจำ (RAM) ประเภทที่ไม่ตรงกับที่เมนบอร์ดรองรับ (DDR4 vs DDR5) ซึ่งก่อให้เกิดความเสียหายทางทรัพย์สินและเวลา
3. **ความล้มเหลวในการวิเคราะห์จุดอุดตันของระบบเดิม (Bottleneck Analysis failure):** เมื่อผู้ใช้ทั่วไปต้องการอัปเกรดคอมพิวเตอร์เดิมที่ทำงานล่าช้า มักจะไม่สามารถระบุชิ้นส่วนที่เป็นคอขวด (Bottleneck) ที่แท้จริงของระบบได้ ทำให้เสียค่าใช้จ่ายในการอัปเกรดชิ้นส่วนที่ไม่ตรงจุด (เช่น การเพิ่มขนาดแรม แต่คอขวดจริงอยู่ที่ฮาร์ดดิสก์แบบจานหมุน HDD)
4. **ความยุ่งยากและเสียเวลาในการค้นหาข้อมูลและเปรียบเทียบ:** การหาข้อมูลรายละเอียดฮาร์ดแวร์เพื่อประกอบการตัดสินใจมีปริมาณมากและกระจัดกระจาย ขาดระบบอัตโนมัติที่เป็นกลาง ปลอดภัย และสามารถโต้ตอบตอบคำถามเชิงลึกในรูปแบบภาษาธรรมชาติ (Natural Language) ได้ตลอด 24 ชั่วโมง

---

## 2. สาเหตุของปัญหา (Root Causes)
1. **การพัฒนาทางเทคโนโลยีที่รวดเร็ว:** ตลาดอุปกรณ์คอมพิวเตอร์มีการเปิดตัวรุ่นและเทคโนโลยีใหม่ตลอดเวลา มีชุดคำศัพท์เฉพาะทางเชิงวิศวกรรมจำนวนมาก เช่น TDP (Thermal Design Power), RAM Bus, Form Factor, Core/Thread, VRAM ซึ่งยากต่อการทำความเข้าใจของผู้บริโภคทั่วไป
2. **ความต้องการโครงสร้างสถาปัตยกรรมตรวจสอบสิทธิ์:** ขาดแคลนระบบการประมวลผลตรรกะแบบผสมผสาน (Hybrid Logic) ที่รวมเอาสมองการสื่อสารแชทบอทเข้ากับกลไกคำนวณข้อจำกัดฮาร์ดแวร์ (Hardware Constraints Checking Rules)
3. **การเข้าถึงช่องทางให้คำปรึกษาที่จำกัด:** การให้คำแนะนำจัดสเปคแบบเดิมอาศัยพนักงานขายหน้าร้าน ซึ่งอาจมีข้อจำกัดด้านเวลา หรือปัญหาเรื่องความลำเอียงในการแนะนำแบรนด์เพื่อหวังผลกำไรเชิงพาณิชย์

---

## 3. ผลกระทบ (Impacts)
1. **การสูญเสียทางการเงินของผู้บริโภค:** ผู้ใช้อาจซื้อเครื่องคอมพิวเตอร์ที่มีราคาแพงเกินความจำเป็น หรือซื้อสเปคที่ไม่ตรงกับงานที่ใช้จริง
2. **ความสูญเสียทางกายภาพต่ออุปกรณ์:** การจัดซื้อชิ้นส่วนที่เข้ากันไม่ได้และพยายามฝืนประกอบ อาจทำให้อุปกรณ์อิเล็กทรอนิกส์ชำรุดเสียหายและไม่สามารถเคลมประกันได้
3. **ความล้มเหลวในการปรับปรุงประสิทธิภาพ:** คอมพิวเตอร์เดิมได้รับการอัปเกรดแต่อาจทำงานอืดล่าช้าเท่าเดิมเนื่องจากการวิเคราะห์ปัญหาไม่ตรงคอขวด (Bottleneck)
4. **ประสบการณ์เชิงลบต่อระบบเทคโนโลยีสารสนเทศ:** ผู้ใช้งานขาดความมั่นใจในการจัดซื้อและบำรุงรักษาคอมพิวเตอร์ด้วยตนเอง

---

## 4. วัตถุประสงค์ (Project Objectives)
1. เพื่อออกแบบและพัฒนาแชทบอทอัจฉริยะ (SpecFlow) สำหรับให้บริการจัดสเปคคอมพิวเตอร์ประกอบจำนวน 8 ชิ้นส่วนหลัก ภายใต้เงื่อนไขงบประมาณและการนำไปใช้งานของผู้ใช้ผ่านทางโปรแกรมประยุกต์ LINE
2. เพื่อสร้างระบบตรวจสอบและคัดกรองความเข้ากันได้ทางกายภาพของอุปกรณ์ (Hardware Compatibility Validation Rules) เพื่อรับประกันความปลอดภัยของสเปคที่แนะนำ 100%
3. เพื่อพัฒนาระบบแนะนำและประเมินผลการอัปเกรดคอมพิวเตอร์เดิม (Upgrade Advisor) พร้อมวิเคราะห์และชี้เป้าปัญหาคอขวดของระบบ (CPU/GPU Bottleneck)
4. เพื่อพัฒนาคลังความรู้โต้ตอบ (FAQ) และแนวทางเพิ่มประสิทธิภาพคอมพิวเตอร์ผ่านซอฟต์แวร์เบื้องต้น เพื่อให้ความรู้แก่ผู้ใช้งานทั่วไป
5. เพื่อพัฒนาระบบหลังบ้านสำหรับเก็บข้อมูลประวัติการทำรายการและประมวลผลออกมาเป็นรายงานเชิงสถิติ (Analytics Dashboard API) เพื่อการวิเคราะห์แนวโน้มตลาดและความต้องการ

---

## 5. ขอบเขตระบบ (System Scope)

### 5.1. ขอบเขตด้านการทำธุรกรรมและโต้ตอบของผู้ใช้งาน (End User Scope)
1. **ระบบประมวลผลข้อความภาษาไทยธรรมชาติตามช่องทางแชท LINE:**
   * รองรับข้อความทักทาย สอบถามข้อมูลจัดสเปค แนะนำอัปเกรด และคำถาม FAQ ด้านเทคนิค
   * รองรับการประมวลผลตัดคำภาษาไทย แก้ไขการพิมพ์สะกดผิดของชื่ออุปกรณ์ไอที และตัดคำฟุ่มเฟือยออกอัตโนมัติ
2. **ระบบจัดสเปคคอมพิวเตอร์ (Smart PC Builder):**
   * จัดหาอุปกรณ์คอมพิวเตอร์ครบเซ็ตจำนวน 8 ชิ้นส่วน: CPU, Motherboard, RAM, GPU, Storage, Power Supply (PSU), Case, และ Cooler
   * จัดสรรงบประมาณ 2 เลเยอร์ ( Layer 1: สัดส่วนงบประมาณตามรูปแบบงาน และ Layer 2: การตรวจสอบความเข้ากันได้และจับคู่ Socket/ประเภท RAM/กำลังไฟวัตต์/ขนาดเมนบอร์ดกับเคส)
   * รองรับลักษณะงาน 3 หมวด: เล่นเกม (Gaming), ตัดต่อวิดีโอ/กราฟิก (Editing), และงานออฟฟิศเอกสาร (Office)
   * รองรับความต้องการจัดแบบ "เผื่ออัปเกรดในอนาคต (Future Upgrade)" โดยจะคัดเลือกเฉพาะเมนบอร์ด Socket ล่าสุด (AM5, LGA1700) และใช้หน่วยความจำ RAM DDR5 เท่านั้น
   * คัดค้านและปฏิเสธหากงบประมาณที่ผู้ใช้ระบุมาน้อยกว่า 10,000 บาท
3. **ระบบวิเคราะห์การอัปเกรด (Upgrade Advisor):**
   * รับข้อมูลรายละเอียดสเปคเดิมและเป้าหมายการนำไปใช้งาน จากนั้นวิเคราะห์หาจุดบกพร่องรายชิ้นส่วน (แรมต่ำกว่ามาตรฐาน, ฮาร์ดดิสก์แบบจานหมุน HDD, หรือการ์ดจอแสดงผลตระกูลเก่า)
   * แสดงข้อคิดเห็นแจ้งเตือนภาวะคอขวดระบบแบบเจาะจง (CPU Bottleneck / GPU Bottleneck)
   * แนะนำชิ้นส่วนที่ควรอัปเกรดพร้อมประเมินยอดรวมงบประมาณค่าใช้จ่ายโดยประมาณ
4. **ระบบบริการตอบคำถามเทคนิคเบื้องต้น (FAQ & Software Tuning):**
   * ให้คำอธิบายข้อมูลเทคนิคเกี่ยวกับ CPU, GPU, RAM, SSD vs HDD ในการ์ดแสดงผล LINE Flex Message ที่มีรายละเอียดครบถ้วน
   * แนะนำวิธีการจูนซอฟต์แวร์ระบบปฏิบัติการให้ทำงานลื่นขึ้น (Startup block, Disk Cleanup, Game Mode)
5. **การแสดงผลทางหน้าจอ:**
   * แสดงผลลัพธ์ผ่าน LINE Flex Message การ์ดแสดงรายการสวยงาม มีราคาระบุแยกชิ้น ราคารวม คำเตือน Disclaimer และรูปภาพปุ่มสั่งการ

### 5.2. ขอบเขตด้านผู้บริหารระบบและนักพัฒนา (Admin & Developer Scope)
1. **ระบบฐานข้อมูลฮาร์ดแวร์คลังสินค้า:** 
   * รองรับการปรับเปลี่ยนข้อมูลชิ้นส่วนใน `hardware_db.json` หรือเขียนโค้ดเพิ่มผ่านสคริปต์ `generate_hardware_db.py`
2. **ระบบเตรียมฝึกสอนข้อมูล AI Chatbot:** 
   * มีสคริปต์ช่วยเหลือ `preprocess_nlu_data.py` ในการทำความสะอาดและจัดระเบียบคำใน `nlu.yml` เพื่อรักษาความถูกต้องของ Entity markup ก่อนส่งเทรน
3. **ระบบสรุปรายงานสถิติใช้งาน (Analytics Report):**
   * รองรับคำสั่งเรียกอ่านและวิเคราะห์ประวัติการค้นหาทั้งหมดผ่านสคริปต์ `analytics_report.py` สรุปรายงานจำนวนการใช้งานรายสัปดาห์ รายเดือน ความต้องการยอดนิยม และช่วงงบประมาณเฉลี่ย

### 5.3. ข้อจำกัดของระบบ (System Constraints)
1. ระบบสนทนารองรับเฉพาะภาษาไทยเป็นหลัก
2. ข้อมูลราคาและประเภทฮาร์ดแวร์อ้างอิงตามฐานข้อมูลจำลอง (Mock Database ในรูปแบบ JSON) ซึ่งต้องได้รับการอัปเดตแบบ Manual โดยนักพัฒนา
3. ระบบจะทำงานและตอบสนองก็ต่อเมื่อเปิดการรันคำสั่งบริการ Rasa Server, Action Server และมี Ngrok Tunnel เปิดเชื่อมต่อกับ LINE Platform

## 6. ข้อกำหนดเชิงฟังก์ชัน (Functional Requirements)

เอกสารระบุข้อกำหนดเชิงฟังก์ชัน (Functional Requirements - FR) แบ่งออกเป็นรหัสตั้งแต่ **FR1 ถึง FR8** ดังรายละเอียดในตารางต่อไปนี้:

| รหัส | ชื่อฟังก์ชัน | รายละเอียดการทำงานของระบบ (System Behavior) |
| :--- | :--- | :--- |
| **FR1** | การเชื่อมต่อ LINE Gateway และตรวจสอบลายเซ็น | ระบบต้องสามารถรับ HTTP POST request จาก LINE webhook ตรวจลายเซ็น `X-Line-Signature` ด้วย `channel_secret` และส่งผ่านข้อมูลแชทผู้ใช้สู่ตัวประมวลผล |
| **FR2** | การทำความสะอาดภาษาไทยและการปรับแต่งคำสะกดผิด | ระบบต้องดึงข้อความแชทมาทำความสะอาด ลบคำสร้อย/คำฟุ่มเฟือยภาษาไทยที่ไม่ได้ส่งผลต่อการวิเคราะห์ (เช่น ครับ, ค่ะ, อยากได้) และแปลงคำพิมพ์ผิดยอดนิยมเป็นคำกลาง เช่น "เมนบอด" $\rightarrow$ "motherboard", "แรม" $\rightarrow$ "ram" ก่อนส่งให้โมเดล NLU |
| **FR3** | การจำแนกความต้องการและสกัดตัวแปร (NLU Classifier) | ระบบต้องใช้โมเดล DIETClassifier ร่วมกับ Custom ThaiTokenizer ในการประเมิน Intent ของประโยค (จัดสเปค, อัปเกรด, ถามข้อมูลชิ้นส่วน) และสกัด Entity (งบประมาณ, วัตถุประสงค์งาน, ชิ้นส่วนไอที) ออกมาได้อย่างแม่นยำ |
| **FR4** | การนำบทสนทนาโต้ตอบผ่านแบบฟอร์ม (Form Management) | ระบบต้องควบคุมบทสนทนาผ่าน Forms (`build_pc_form` / `upgrade_pc_form`) หากพบว่าช่องข้อมูลที่จำเป็น (Slots) ยังไม่มีค่า ระบบต้องส่งคำถามพร้อมการ์ดปุ่ม Quick Reply เพื่อให้ผู้ใช้งานตอบกลับข้อมูลนั้นๆ จนกว่าจะครบถ้วน |
| **FR5** | การจัดสรรสเปคและตรวจสอบความเข้ากันได้ (Spec Calculation) | ระบบต้องคำนวณแบ่งสเปคออกเป็น 8 ชิ้นส่วนตามกฎแบ่งร้อยละงบประมาณ และผ่านการกรองกฎความเข้ากันได้ (Socket, RAM Type, Case size, Wattage และ CPU Cooler support) และหากเลือกแบบเผื่ออัปเกรดต้องเลือกเฉพาะ Socket ล่าสุดและ RAM DDR5 |
| **FR6** | การประเมินปัญหาประสิทธิภาพเดิม (Upgrade Analysis) | ระบบต้องวิเคราะห์ข้อความบรรยายสเปคเดิมของผู้ใช้ ระบุชิ้นส่วนที่เสื่อมสภาพ คาดคะเนภาวะคอขวดและชี้แจงสาเหตุ เช่น CPU Bottleneck หรือ GPU Bottleneck และเสนอรายการอัปเกรดอุปกรณ์ใหม่พร้อมราคารวมประเมิน |
| **FR7** | การตอบข้อคำถามคลังความรู้และการปรับความเร็วระบบ (FAQ System) | ระบบต้องตอบกลับข้อความถามความรู้ด้านฮาร์ดแวร์พื้นฐาน และให้คำแนะนำทางเทคนิคในการตั้งค่าเพิ่มความเร็วในส่วนของซอฟต์แวร์ปฏิบัติการ Windows |
| **FR8** | การบันทึกและสร้างรายงานวิเคราะห์เชิงลึก (Database Logging & Analytical Reports) | ระบบต้องเก็บล็อกประวัติผู้ใช้งานรายรายการลง SQLite database `analytics.db` และผู้ดูแลระบบต้องสามารถรันรายงานสถิติวิจัย (รายสัปดาห์, รายเดือน, ส่วนงบประมาณยอดนิยม และรูปแบบประเภทงาน) ได้ทางหน้าคอนโซลคำสั่ง |

## 7. ข้อกำหนดที่ไม่ได้เกี่ยวข้องกับฟังก์ชันหลัก (Non-functional Requirements)

เอกสารระบุข้อกำหนดที่ไม่ได้เกี่ยวข้องกับฟังก์ชันหลัก (Non-functional Requirements - NFR) แบ่งเป็นรหัส **NFR1 ถึง NFR5** ดังตารางต่อไปนี้:

| รหัส | คุณลักษณะที่ต้องการ | มาตรฐานการวิเคราะห์และข้อกำหนดการทำงาน (Expected Target) |
| :--- | :--- | :--- |
| **NFR1** | **ความปลอดภัยของระบบ (Security & Integrity)** | การรับส่งข้อมูลผ่าน Webhook เชื่อมต่อ LINE API ต้องเข้ารหัสผ่าน SSL/HTTPS และตรวจสอบลายเซ็น SHA256 Signature ด้วยคีย์เฉพาะคู่เสมอเพื่อสกัดความถูกต้อง ป้องกันการโจมตีปลอมแปลงข้อมูล |
| **NFR2** | **ความถูกต้องของตรรกะความเข้ากันได้ (Physical Compatibility Accuracy)** | กฎตรวจสอบความเข้ากันได้ทางกายภาพคอมพิวเตอร์ (Socket, TDP, RAM Type, Form Factor) ต้องมีความถูกต้อง 100% ตามมาตรฐานโรงงานผู้ผลิตฮาร์ดแวร์ที่กำหนดในระบบ เพื่อไม่ให้สเปคที่ประกอบเสียหาย |
| **NFR3** | **ความถูกต้องแม่นยำของระบบสนทนา (NLU Model Accuracy)** | โมเดล Rasa NLU ที่ผ่านกระบวนการฝึกฝนและตัดคำภาษาไทยแล้ว จะต้องผ่านการทดสอบ (`rasa test nlu`) ได้ค่าความถูกต้อง (Accuracy/F1-Score) เฉลี่ยมากกว่า 90% ในข้อมูลชุดทดสอบการใช้งานทั่วไป |
| **NFR4** | **ประสิทธิภาพความรวดเร็วการตอบกลับ (Response Latency)** | ระยะเวลาในการคำนวณและตอบสนองข้อความกลับไปยัง LINE Gateway ตั้งแต่บอทได้รับ Webhook ต้องใช้เวลาเฉลี่ย (Processing time) ไม่เกิน 3.0 วินาที เพื่อสร้างประสบการณ์การโต้ตอบที่ดี |
| **NFR5** | **ความทนทานต่อข้อผิดพลาด (Fault Tolerance & Data Integrity)** | ระบบตรรกะหลังบ้านจะต้องทนทานต่อข้อความว่างเปล่า (Null/Empty String) หรืออินพุตที่อยู่นอกระบบขอบเขต และ SQLite Database ต้องเริ่มกลไกรื้อสร้างตารางเริ่มต้นอัตโนมัติหากตรวจพบว่าไม่มีตารางหรือฐานข้อมูลว่างเปล่า |
